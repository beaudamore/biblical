# SFT Data Generation

How the supervised-fine-tuning dataset is built. Covers Q&A generation, voice-quality enforcement, and the raw-text continuation augmentation that distinguishes v2.

The two-stage architecture: **(a)** persona-aware Q&A generated by a large thinking model, gated on voice-uniqueness metrics, then **(b)** free continuation chunks blended in at 60/40 to teach the cadence of biblical prose itself, not just the ability to answer questions in it.

Notebook: [`notebooks/datagen/biblical_datagen_v2_sft.ipynb`](../notebooks/datagen/biblical_datagen_v2_sft.ipynb)

---


Generation is **OpenAI-compatible**, so swapping the endpoint to a local vLLM, DeepInfra, or any other provider is one line.

## 2. Persona prompts — the voice-locking layer

Every persona is defined as a structured block:

```python
PERSONA_METADATA = {
    "amos": {
        "identity":     "a shepherd and fig farmer from Tekoa, called by God to prophesy ...",
        "voice_notes":  "Blunt rural speech. Agricultural imagery — plowing, harvest, locusts, figs ...",
        "kjv_exemplars": [
            "Thus saith the LORD; For three transgressions of Damascus, ...",
            "But let judgment run down as waters, and righteousness as a mighty stream.",
            ...
        ],
    },
    ...
}
```

Three components, each pulling its own weight:

| Component | What it gives the model |
| --- | --- |
| **identity** | A first-person grounding (`"I am ..."` framed) — keeps answers anchored in actual biographical events from the text rather than abstract theology. |
| **voice_notes** | Explicit prose-style guidance: *Blunt rural speech, agricultural imagery, thundering declarations.* This shapes the **register**, not just the topic. |
| **kjv_exemplars** | Five direct KJV quotes attributed to the persona. These are the strongest cadence anchor the model gets — short enough that the LLM treats them as style cues rather than content to paraphrase. |

These three fields are wrapped into a single ShareGPT system prompt at conversation-assembly time. That same system prompt is what the trained LoRA expects at inference, so the persona definition is part of the deployed contract.

### Banned openers — global safety net

Every model trained on instruct data has favorite generic openers. The pipeline defines them up front:

```python
BANNED_OPENERS = [
    "The weight of", "My friend,", "The memory of", "The memories of",
    "My child,", "My brother,", "My sister,", "My son,",
    "The moment", "I remember", "I recall",
    "You see,", "Ah,", "Brother,", "Friend,",
    "Let me tell you,", "Well,", "You know,",
]
```

Two enforcement points:

1. **At generation time** — `is_template_answer(answer)` runs before each answer is written to disk. A match triggers a regenerate-with-different-temperature retry inside the answer call (see the `_single_answer_call` retry loop). This catches ~30% of the would-be contaminated outputs without burning extra tokens.
2. **At quality-gate time** — the surviving template fraction per persona is measured and reported. If global contamination exceeds 30%, the gate **fails closed** and refuses to assemble the dataset (see §4 below).

This is a cheap, deterministic guardrail that keeps the most obvious "Llama3-isms" out of the training data without needing a learned classifier.

## 3. Question generation — three rounds, three angles

For every chunk, three different system prompts are rotated:

| Round | Style |
| --- | --- |
| **1. Factual / interpretive** | who/what/when/where about specific events; *why did you do that, what did it mean to you*. Anchored in passage details. |
| **2. Application** | "Based on what you went through, how should I handle ...". Connects the persona's specific events to a parallel modern situation. |
| **3. Reflective** | "What were you feeling when ... happened?", "Looking back on ..., what would you tell someone who doubts?". Aimed at first-person emotional disclosure. |

Each round generates `QUESTIONS_PER_CHUNK = 5` questions per chunk, so a chunk produces 15 Q&A pairs total.

All three prompts share the same anti-template constraints:
- "Frame as if speaking DIRECTLY to {persona_name} — use 'you' and reference their specific experiences."
- "Reference specific details from the passage (names, places, events) — NOT generic theology."
- "Do NOT say 'the text' or 'the passage'."
- JSON-only response (`response_format={"type": "json_object"}`).

Questions and answers are produced by separate API calls. Questions use `temperature=0.9` (we want diversity in question shape); answers use `temperature=0.7` (we want voice consistency). Both calls are wrapped in `_api_call_with_timeout(coro, timeout_secs=120)` so a hung request can't permanently hold the concurrency semaphore.

### Streaming + resume

Each persona's per-chunk Q&A is appended to `data/training-data/biblical_persona_v2/per_persona/<persona>.jsonl` **as it completes** — the notebook never holds all generated data in RAM. Re-running the cell skips chunks that already have their full Q&A budget on disk, so a crashed or interrupted run resumes idempotently rather than re-billing the API.

## 4. Voice-differentiation quality gate

Section 5b of the notebook. Runs **before** any conversation is assembled, blocks the assembly cell if it fails.

For each persona's `<persona>.jsonl`:

1. **Template contamination %** — fraction of answers whose first ~16 lowercase chars match a banned opener.
2. **4-gram opener uniqueness** — extract the first 4 words of every answer, count global occurrences across all 26 personas, then compute the per-persona "% unique openers" (how many of that persona's openers appear nowhere else).

Per-persona report:

```text
Persona         Total  Template  Contam%  Status
amos             750         3     0.4%   ✓ PASS
peter            750        12     1.6%   ✓ PASS
joshua           750        45     6.0%   ✓ PASS
...
GLOBAL         19500       180     0.9%
```

Cross-persona report:

```text
Top 10 most repeated opening 4-grams:
   38x across 24 personas: "I look back and"
   29x across 19 personas: "When I was a"
   ...

Per-persona opener uniqueness:
  amos       72% unique openers (540/750)
  david      89% unique openers (668/750)
  ...
```

Decision logic:

| Global contamination | Decision |
| --- | --- |
| >30% | `QUALITY_GATE_PASSED = False` — assembly is **blocked**. Operator must delete failing personas' `per_persona/*.jsonl` and re-run generation. |
| 15–30% | `QUALITY_GATE_PASSED = True` with a `WARN` printout. |
| <15% | `QUALITY_GATE_PASSED = True`, no warning. |

The 4-gram report is informational — there's no automatic threshold on it, but a top-10 dominated by one persona is a strong signal that something is off (e.g., a single template prompt accidentally got reused for multiple personas).

## 5. Conversation assembly

Q&A pairs are grouped into multi-turn ShareGPT conversations:

- `TURNS_PER_CONVERSATION = 4` — each conversation contains 4 Q&A pairs (8 turns plus 1 system) from the **same persona**.
- The system turn carries the full persona prompt (identity + voice_notes + KJV exemplars).
- Pairs are shuffled within a persona before grouping, so each conversation contains a mix of factual / application / reflective questions rather than three rounds in fixed order.
- After all personas are grouped, the entire JSONL is shuffled (Python `random.shuffle` + a final `shuf -o` pass for extra randomness).

Output: `data/training-data/biblical_persona_v2/biblical_personas_sharegpt.jsonl` (~9,700 conversations).

## 6. Continuation augmentation — the v2 addition

The model has now learned to *answer* questions in 26 distinct voices. But it hasn't directly absorbed the cadence of Biblical prose itself — the parallelism structure of Hebrew poetry, the chained clauses of Pauline argumentation, the legal-formula rhythm of Deuteronomy.

Section 8 of the notebook fixes that with a free, no-API-call augmentation pass. For each persona's source text:

1. Token-aware chunk into 500-tok blocks (see [data-pipeline.md §4](data-pipeline.md#4-token-aware-chunking--continuation-pipeline)).
2. Split each block into a 60-token **seed** and ~440-token **completion**, snapped to a clause boundary.
3. Pick a random instruction from `CONTINUATION_INSTRUCTIONS` (see below).
4. Emit a ShareGPT conversation:

   ```json
   {
     "conversations": [
       {"from": "system", "value": "<persona's full system prompt>"},
       {"from": "human",  "value": "<instruction>\n\n<seed>"},
       {"from": "gpt",    "value": "<completion>"}
     ],
     "data_type": "continuation"
   }
   ```

Continuation instruction templates (sampled per chunk):

```python
CONTINUATION_INSTRUCTIONS = [
    "Continue writing in this voice and style, carrying forward the themes and language:",
    "Continue this passage, maintaining the same tone, vocabulary, and cadence:",
    "Write what comes next, staying true to the voice and spirit of this text:",
    "Carry on from where this passage leaves off, preserving the distinctive style:",
    "Continue this text naturally, as if you were the original speaker:",
]
```

Each persona produces a continuation `.jsonl` in `data/training-data/biblical_persona_v2/augmented/continuation/<persona>_continuation.jsonl`. Resume-by-skip is on: if the file exists and is non-empty, that persona is skipped.

The training signal here is **pure prose cadence**: there's no LLM-generated answer, just the raw KJV (or other source) text serving as ground truth for "what comes after this seed." That's why this pass costs zero API tokens.

## 7. Final blend — section 9

```python
TARGET_BLEND = {
    "qa":           0.60,
    "continuation": 0.40,
    # "cot":         0.20,    # scaffolded, not active
    # "instruction": 0.10,    # scaffolded, not active
}
```

Q&A is the anchor: total dataset size is computed as `qa_count / 0.60`. Each pool is then up-sampled (with cycling + remainder) or down-sampled (random subset) to hit its target fraction.

Output: `data/training-data/biblical_persona_v2/biblical_personas_combined_sharegpt.jsonl`. Final shuffle uses `random.shuffle()` followed by a `shuf -o <file> <file>` external call so the random state of the Python session doesn't bias chunk ordering.

A `data_type` field is preserved on every entry so the trainer (or post-hoc analysis) can split Q&A vs continuation losses if desired. The trainer itself ignores extra keys — including `data_type` is harmless and has been useful for debugging.

## 8. Scaffolded augmentations (not yet active)

The notebook has feature-flag gates for two more data types that aren't generated by default:

- **`ENABLE_COT`** — chain-of-thought reasoning passes using `<think>` tags with a thinking model. Targeted at theological reasoning chains (historical context → meaning → application). Requires API calls.
- **`ENABLE_INSTRUCTIONS`** — broader task templates beyond Q&A: *"Summarize this parable,"* *"Explain this prophecy as Paul would,"* *"Write a prayer about ..."*. Also API-paid.

When enabled, the blend ratios shift to `40% Q&A / 30% continuation / 20% CoT / 10% instruction`. The merge logic in section 9 already handles arbitrary `data_pools`, so adding these is mostly a matter of generating the per-persona JSONLs and adding them to the `TARGET_BLEND` dict.

## 9. Ancillary pipelines — Augustine and Liguori

`biblical_datagen_augustine.ipynb` and `biblical_datagen_liguori.ipynb` are **structural twins** of the v2 SFT notebook with different source corpora and persona definitions:

- **Augustine** uses 10 Augustinian works (Confessions, City of God I+II, Donatist, Soliloquies, etc.) with multi-mode "voice" definitions (theologian, pastor, autobiographer). Output: `data/training-data/augustine_persona/`.
- **Liguori** uses 6 of his devotional works (Maxims, Way of the Cross, Visits to the Most Holy Sacrament, etc.) with a single Alphonsus persona. Output: `data/training-data/liguori_persona/`.

Both run the same 3-round Q&A loop, banned-openers list, and quality gate. Neither runs the continuation-augmentation step (it would help — adding it is a future unification task).

These ancillaries are kept separate so a single LoRA can be trained on a single saint without contamination from cross-persona data.
